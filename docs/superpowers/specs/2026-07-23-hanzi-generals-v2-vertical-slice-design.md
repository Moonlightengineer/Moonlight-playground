# 字陣無雙 v2｜群雄遠征 Vertical Slice 設計規格

- 狀態：Design approved；written spec pending Owner review
- 版本：v0.1
- 日期：2026-07-23
- 目標平台：Mobile-first Web／GitHub Pages
- 實作狀態：未開始

## 1. 目的

將現有《字陣無雙》由固定 3×3、十波守城小遊戲，驗證性升級為：

> 中文字牌合成 × 可成長戰陣 × 流派構築 × 半自動戰鬥 × 短局 Roguelite

Vertical Slice 唔追求完整內容量，而係用一局六戰、約 15–20 分鐘嘅短遠征，驗證以下假設：

1. 棋盤擴展會唔會真正改變部署選擇。
2. 字牌構築會唔會形成策略，而唔係只靠抽運。
3. 敵人意圖會唔會令失敗原因清晰。
4. 有限軍令會唔會帶來有意義嘅戰鬥介入。
5. 玩家完成一局後，會唔會願意以另一套流派再玩。

任何「好玩、平衡、重玩性高」結論都只可由真人試玩支持；自動測試只證明規則正確。

## 2. 發布及兼容策略

採用 Classic 與 v2 並行方案。

### Classic

- 現有入口維持：`/games/hanzi-generals/`
- 現有十波模式保持可玩。
- 現有壓縮分段封裝不作 v2 基礎。
- v2 開發不可破壞 Classic 建置、網址或存檔。

### v2 隱藏測試版

- 新入口：`/games/hanzi-generals/v2/`
- 不加入根目錄 `projects.json`。
- Playground 首頁不顯示 v2 卡片。
- 知道網址嘅測試者先可以進入。
- 通過 Vertical Slice 試玩 Gate 後，先決定是否建立模式選擇頁及公開入口。

v2 必須保存正常可讀、可測試嘅 HTML、CSS、JavaScript 原始檔，不使用 Classic 嘅 gzip＋Base64 分段封裝模式。

## 3. 目標玩家與體驗

### 目標玩家

- 喜歡輕策略、合成、牌庫構築或自動戰鬥。
- 對三國人物有基本印象，但不需要熟悉歷史。
- 主要使用手機，每局可投入約 15–20 分鐘。
- 希望操作易明，但可以透過牌庫、位置及反制逐步掌握深度。

### 玩家幻想

由散亂中文字牌中招募名將、建立戰陣，閱讀敵軍部署後以有限軍令救場，完成一場屬於自己嘅短遠征。

### 設計支柱

1. **文字有用途**：抽到嘅字應該可以構成武將、兵種或牌庫調整決策，避免純垃圾牌。
2. **位置比純數值重要**：射程、攻擊方向、路線及陣形要直接影響結果。
3. **每局形成流派**：獎勵應改變之後嘅選擇，而唔係只提供小量百分比加成。
4. **失敗可解釋**：玩家要睇得出邊一路失守、邊個能力未處理，同下一局可改乜。
5. **手機第一**：棋盤、主要操作、敵人意圖及即時後果優先顯示。

## 4. 一局遠征結構

每局固定完成六場戰鬥；第三戰有路線分支，但玩家每局只打其中一條。

1. **第一戰｜教學戰**：放字、合成、查看攻擊範圍、使用一次軍令。
2. **第二戰｜盾陣壓境**：引入盾兵及集火需求；戰後提供一次牌庫整理／刪牌。
3. **第三戰｜路線選擇**：
   - 安全路線：較弱敵軍；戰後獲得張飛完整配方包及較多城牆修復。
   - 危險路線：旗手＋弩兵組合；戰後獲得諸葛亮完整配方包及較強構築獎勵。
4. **第四戰｜重騎預警**：引入倒數衝鋒及路線防守；戰後必須選擇 4×3 或 3×4 擴陣。
5. **第五戰｜精英混編**：要求玩家同時處理保護、遠射及突擊威脅；戰後必須為一名已招募武將選擇進化。
6. **第六戰｜華雄 Boss**：多階段、換線、低血召喚重騎。

遠征中固定出現：

- 一次分支路線選擇。
- 一次牌庫整理／刪牌機會。
- 一次武將進化。
- 一次戰陣擴展。
- 至少一次城牆修復選擇。

存檔只在節點完成、獎勵確認及新戰鬥開始前寫入；不提供戰鬥中途讀檔。

## 5. 戰鬥節奏

每場戰鬥分成三個配置與戰鬥交替階段：

1. 抽牌及初步佈陣 → 敵軍前鋒。
2. 補牌及調整 → 敵軍主力。
3. 最後整軍 → 精英／主將。

### 同一場戰鬥內保留

- 棋盤位置。
- 已合成武將及兵種。
- 單位剩餘生命及狀態。
- 已使用軍令點。
- 未用而被保留嘅手牌。

### 戰鬥完結後

- 棋盤、軍營及場上單位清空。
- 所有牌回到遠征牌庫，供下一戰重新抽取。
- 牌庫修改、陣圖選擇、武將進化及遠征獎勵保留。
- 城牆生命跨戰保留，戰後預設回復最大值 15%，但不可超過上限。
- 如玩家選擇修復獎勵，額外回復量由獎勵資料定義。

城牆生命降至 0，遠征立即失敗。結算畫面必須列出最後失守路線、主要傷害來源及未處理特殊能力。

## 6. 棋盤及陣圖演化

### 座標規則

棋盤尺寸統一以「欄 × 列」表示：

- 欄：左右路線數量。
- 列：前後縱深。

### 起始及擴展

- 起始棋盤：3×3。
- 遠征中只擴展一次。
- 玩家二選一：
  - **4×3 側翼擴展**：增加第四條路線，適合換線、橫掃及分散敵軍。
  - **3×4 縱深擴展**：增加一列，適合前排防守及後排遠射。
- Vertical Slice 不支援 4×4、4×5 或任意形狀棋盤。

### 位置規則

- 字牌只可以放入合法空格或軍營格。
- 點選字牌後，只高亮合法位置。
- 武將顯示攻擊方向、射程及目前鎖定目標。
- 擴陣後，既有資料模型必須可以使用相同座標與目標選擇規則，不為每種尺寸寫獨立戰鬥邏輯。

## 7. 抽牌、牌庫、部署區及軍營

### 抽牌

- 每個配置階段抽至 5 張手牌。
- 每個階段結束時最多保留 2 張未用字牌。
- 每場戰鬥有 1 次免費重抽；玩家可以保留最多 2 張已選手牌，其他手牌放入棄牌堆後補抽至 5 張。
- 未使用而不保留嘅手牌進入棄牌堆。
- 抽牌堆耗盡時，棄牌堆按該局種子重新洗牌。
- UI 顯示抽牌堆、棄牌堆及部署區牌數量。

### 起始牌庫

起始牌庫包含 Classic 四名武將嘅完整字組：

- 黃、忠
- 趙、雲
- 關、羽
- 呂、布

另包含基本兵種字組：

- 弓、兵
- 盾、兵

牌庫初始共 12 張。重複「兵」字係刻意設計，令弓兵與盾兵存在競爭。

### 牌嘅生命週期

- 成功合成單位後，構成該單位嘅字牌移到「部署區」，唔會立即進棄牌堆。
- 單位仍然存活時，相關字牌不可再次抽取。
- 單位死亡後，相關字牌進棄牌堆，之後有機會重新抽取及重建。
- 每名具名武將同一時間只可有一個存活實體。
- 如同名武將仍存活，完成相同配方嘅操作會被拒絕並顯示原因。
- 弓兵及盾兵不屬具名武將；如牌庫有足夠配方，可以部署多個。

### 軍營

- 棋盤外預設設兩格軍營。
- 字牌可以放入軍營暫存。
- 軍營內形成完整配方時，玩家必須立即選擇合法棋盤位置部署；未有合法位置則不可確認合成。
- 合成部署完成後，軍營格即時清空，相關字牌進部署區。
- 軍營滿時不可再放牌。
- 軍營唔提供額外抽牌或跨戰永久儲存；戰鬥結束時清空。

呢個混合系統保留「文字直接砌戰陣」特色，同時減少差一個字時阻塞棋盤。

## 8. 配方、兵種及六名武將

### 配方

- 黃＋忠 → 黃忠
- 趙＋雲 → 趙雲
- 關＋羽 → 關羽
- 呂＋布 → 呂布
- 張＋飛 → 張飛
- 諸＋葛＋亮 → 諸葛亮
- 弓＋兵 → 弓兵
- 盾＋兵 → 盾兵

張飛與諸葛亮唔放入起始牌庫，只會按第三戰分支以完整配方包加入，避免三字武將需要連續取得三個獨立獎勵先可使用。

### 基本兵種

- **弓兵**：低生命、較低傷害；攻擊同一路線最近敵人，基礎射程為 2 格；無進化。
- **盾兵**：低傷害、高生命；優先阻擋同一路線最接近城牆敵人，並為正後方一格友軍提供減傷；無進化。

兵種用途係補位及支援，唔應取代具名武將作主要流派核心。

### 武將定位及互斥進化

| 武將 | 基本定位 | 進化 A | 進化 B |
|---|---|---|---|
| 黃忠 | 後排遠射 | 神射：極遠單體重擊 | 連弩：快速多段攻擊 |
| 趙雲 | 穿透突擊 | 七進：連續穿透 | 護主：攔截接近城牆敵人 |
| 關羽 | 橫排攻擊 | 青龍：擴大橫掃範圍 | 威震：命中後削弱敵軍 |
| 呂布 | 近身爆發 | 無雙：高額重擊 | 飛將：快速轉線追擊 |
| 張飛 | 防守反擊 | 咆哮：嘲諷一條路線 | 燕人：低生命時增傷 |
| 諸葛亮 | 範圍控制 | 火計：持續燃燒 | 八陣：調整友軍位置或攻擊方向 |

每名武將每局只可選一條進化。Vertical Slice 不支援重置、雙路線同時取得或多級技能樹。

所有傷害、攻速、射程、冷卻與狀態值放入版本化 tuning data；本文定義行為，不宣稱初始數值已平衡。

## 9. 敵軍及意圖

### 四種特殊敵人

| 敵人 | 行為 | 主要反制 |
|---|---|---|
| 盾兵 | 保護同排後方敵軍 | 橫掃、穿透、轉線或先破盾 |
| 旗手 | 持續強化附近敵軍 | 集火或快速突進 |
| 弩兵 | 未接近城牆已可攻擊後排 | 趙雲突進、堅守或優先擊殺 |
| 重騎 | 倒數完成後快速突破一條路線 | 預先佈防、變陣、堅守或提前消滅 |

基本步兵仍可作無特殊能力嘅數量單位。

### 敵人意圖

每個重要敵人必須顯示：

- 下一步移動或攻擊方向。
- 目前目標。
- 特殊能力倒數。
- 強化來源。
- 玩家目前有邊個單位鎖定佢。

每個戰鬥階段開始前顯示簡短路線提示，例如：

- 左路：安全。
- 中路：重騎兩個行動輪後衝鋒。
- 右路：旗手正在強化弩兵。

提示只表達風險，不預測必勝或必敗。

## 10. 半自動戰鬥及軍令

戰鬥引擎採用固定離散行動輪，而唔依賴不穩定即時物理。畫面可以用動畫連接行動輪，並支援暫停、1×及 2×速度。

### 軍令資源

- 每場戰鬥開始有 3 點軍令。
- 三個階段共用同一軍令池。
- 軍令不自然回復，不消耗金幣。
- 戰鬥暫停後可以發出軍令；指令在下一個安全行動輪生效。

### 三種軍令

1. **變陣**：交換兩個相鄰己方單位；不可交換死亡、正執行攻擊或位於非法位置嘅單位。指令於下一個安全行動輪生效。
2. **集火**：指定一名目前可被至少一個友軍攻擊嘅敵人；合法友軍優先攻擊該目標，持續 3 個友軍行動輪或直到目標死亡。
3. **堅守**：指定一條路線；該路友軍及城牆受到傷害降低 40%，持續 2 個敵軍行動輪。

40% 及持續行動輪屬初始 tuning hypothesis，可在不改規則語意下調整。

停用狀態必須顯示原因，例如「軍令不足」、「目標不在合法範圍」或「當前行動未完成」。

## 11. Boss｜華雄

華雄用作 Vertical Slice 最終系統測試，而唔係純高生命敵人。

### 階段一

- 於中央路線推進。
- 每 3 個敵軍行動輪蓄力重擊一次。
- 玩家可透過堅守或變陣應對。

### 階段二

- 生命降至最大值 50% 時觸發一次。
- 轉移到相鄰路線。
- 召喚一隊重騎。
- 取得持續 2 個敵軍行動輪嘅盾兵保護。
- 要求玩家決定集火 Boss、先處理重騎，或調動防線。

Boss 戰必須讓三種軍令都有合理使用機會，但唔強迫唯一解法。

## 12. 戰後獎勵及構築

一般戰鬥後出現三選一；劇本 Gate 使用指定選擇集。獎勵生成器必須確保至少一個選項同目前牌庫或流派相容。

獎勵類型：

- 複製一張現有核心字。
- 移除一張牌。
- 選擇一名已招募武將進化。
- 修復城牆。
- 下一場增加一次重抽。
- 下一場暫時增加一格軍營。
- 取得一次性軍策。
- 取得 4×3 或 3×4 擴陣選擇。

### 一次性軍策

Vertical Slice 只提供兩種，使用後從當局移除：

- **火矢**：指定一條路線；下一個友軍行動輪開始時，該路所有敵人受到一次傷害及短燃燒。
- **急救**：指定一個存活友軍，立即回復最大生命 30%。

獎勵不可全部係純攻擊百分比。每個獎勵要改變牌庫一致性、位置選擇、軍令節奏或敵軍反制。

## 13. UI／UX 及 Game Feel

### 手機畫面層級

1. 當前戰鬥階段、城牆及軍令點。
2. 棋盤及敵軍意圖。
3. 手牌與軍營。
4. 主要可用操作。
5. 牌庫、棄牌、部署區、詳細數值及紀錄。

### 操作規則

- 所有拖放功能都有點按替代。
- 點選字牌後只高亮合法格。
- 可合成字牌顯示連線或共同印章提示。
- 長按／資訊鍵顯示攻擊範圍、技能及進化。
- 主要按鈕最小觸控尺寸 44×44 CSS pixels。
- 重要狀態不可只用顏色表達，需配合圖案、文字或倒數。

### 回饋

- 合成武將：短書法展開＋印章效果。
- 攻擊：清楚方向線及命中回饋。
- 重騎衝鋒：提前警告、路線震動或視覺脈衝。
- 支援減少動態效果；關閉後不影響資訊理解。
- 震動預設可用但必須可關閉。
- Vertical Slice 不依賴大型音樂或外部音效素材先可完成核心體驗。

### 新手教學

第一戰只教五項：

1. 點選及放置字牌。
2. 合成一名武將。
3. 查看攻擊方向。
4. 開始戰鬥階段。
5. 暫停並使用一次軍令。

教學成功標準係玩家完成操作，而唔係只睇完文字。

## 14. 技術設計

### 技術限制

- 純 HTML、CSS、原生 JavaScript ES modules。
- 無後端、登入、API Key、資料庫或雲端同步。
- GitHub Pages 靜態發布。
- `localStorage` 保存測試進度及設定。
- v2 初期不與 Classic 共用 runtime code，降低回歸風險。
- 邏輯測試使用 Node 內置 `node:test`，不引入第三方測試框架。

### 建議結構

```text
games/hanzi-generals/v2/
├─ index.html
├─ styles/
│  └─ game.css
├─ src/
│  ├─ app.js
│  ├─ core/
│  │  ├─ state-machine.js
│  │  ├─ rng.js
│  │  ├─ validation.js
│  │  └─ events.js
│  ├─ board/
│  ├─ deck/
│  ├─ combat/
│  ├─ expedition/
│  ├─ storage/
│  └─ ui/
├─ data/
│  ├─ generals.js
│  ├─ enemies.js
│  ├─ recipes.js
│  ├─ stages.js
│  ├─ rewards.js
│  └─ tuning.js
└─ tests/
```

### 模組責任

- `state-machine`：控制遠征、配置、戰鬥、獎勵、結算狀態轉換。
- `rng`：固定種子抽牌、洗牌、獎勵及敵軍變體；相同種子可重播。
- `validation`：啟動時檢查資料結構、ID、配方引用、關卡引用及必要 tuning。
- `board`：座標、合法位置、相鄰、路線及尺寸擴展。
- `deck`：抽牌、棄牌、保留、重抽、部署區、配方及軍營。
- `combat`：離散行動輪、目標選擇、傷害、狀態、意圖及軍令生效。
- `expedition`：六戰節點、分支、獎勵及城牆狀態。
- `storage`：版本化存檔、設定及錯誤恢復。
- `ui`：只根據公開 state render，透過明確 action 呼叫修改遊戲狀態。

資料檔只描述內容；引擎不可按武將或關卡名稱硬編碼特殊分支。華雄階段切換亦以通用 Boss phase data 描述。

### 現有建置整合

實作時必須更新 `scripts/build_site.py`：

- 將 `games/hanzi-generals/v2/` 正常遞迴複製到 `_site/games/hanzi-generals/v2/`。
- 驗證 v2 `index.html` 包含固定 marker，例如 `id="hanzi-v2-app"`。
- 繼續重建及驗證 Classic；Classic 驗證失敗時整個 build 必須失敗。
- 不因 v2 隱藏部署而修改 `projects.json`。

GitHub Actions 應加入 Node 版本及 `node --test` 驗證，但部署輸出仍由現有 Python build script 產生。

## 15. 狀態及資料流

1. 載入頁面後驗證資料檔及存檔版本。
2. 建立新遠征或由最近節點恢復。
3. `state-machine` 進入配置階段。
4. 玩家操作轉成 action，由規則層驗證後修改 state。
5. UI 根據新 state 更新合法位置、資源及提示。
6. 戰鬥開始後，`combat` 按固定行動輪產生事件。
7. UI 消費事件作動畫，但動畫不可決定規則結果。
8. 戰鬥完成後由 `expedition` 產生獎勵及下一節點。
9. 節點確認後保存版本化 snapshot。

## 16. 存檔、錯誤處理及恢復

### 本機儲存鍵

- `moonlight.hanzi-generals.v2.save`
- `moonlight.hanzi-generals.v2.settings`

存檔根層必須包含 `schemaVersion: 1`、遊戲版本、種子、當前節點、城牆、牌庫、獎勵、陣圖及進化資料。

### 錯誤處理

- 資料檔缺少必要欄位：停止開始新局，顯示可理解錯誤及版本資訊，不進入半壞狀態。
- 存檔 JSON 損壞或版本不支援：保留設定，提供「重設 v2 測試存檔」按鈕。
- 非法玩家 action：拒絕操作並顯示原因，不靜默失敗。
- 動畫或震動失敗：規則繼續運作，視覺回饋降級。
- 頁面重整：只由最近節點邊界恢復，不嘗試恢復行動輪中間狀態。
- Classic 資產或建置缺失：CI 必須失敗，v2 不可用修改掩蓋既有問題。

## 17. 測試策略

### 自動規則測試

- 所有配方成功及錯誤組合。
- 3×3、4×3、3×4 合法座標與相鄰關係。
- 抽牌、保留 2 張、免費重抽、洗牌、部署區及固定種子。
- 軍營容量、軍營合成及立即部署。
- 具名武將唯一性、死亡後字牌返回棄牌堆及重新召募。
- 弓兵、盾兵行為及多個兵種部署。
- 六名武將目標選擇與進化互斥。
- 四種特殊敵人意圖及反制規則。
- 變陣、集火、堅守合法、持續時間與非法情況。
- 火矢、急救一次性消耗。
- 華雄兩階段轉換。
- 城牆傷害、15% 戰後回復及失敗結算。
- 存檔版本、損壞存檔及重設流程。

### 決定性整合測試

以固定種子建立：

- 完整六戰遠征 smoke test。
- 4×3 路線。
- 3×4 路線。
- 安全分支及張飛配方包。
- 危險分支及諸葛亮配方包。
- 華雄召喚重騎情境。

### UI 驗證

- 320px 寬度無水平溢出。
- 主要流程可只用點按完成。
- 鍵盤焦點次序合理。
- 減少動態效果模式可完成一局。
- 重要狀態不只靠顏色。
- GitHub Pages 子路徑資產全部正常載入。

### 真人試玩 Gate

最少 5 名首次玩家：

- 4／5 無口頭提示完成第一個合成。
- 4／5 無口頭提示開始首個戰鬥階段。
- 4／5 可以解釋至少一個敵人意圖。
- 4／5 失敗後可以指出一個合理改進方法。

最少 10 次完整或接近完整測試局：

- 觀察到最少 3 種實質不同構築方向。
- 4×3 與 3×4 都有人主動選擇，而唔係單一選項壓倒性明顯。
- 軍令有被保留、提早使用及緊急使用等不同決策，而唔係固定時機按同一個掣。
- 記錄完成後是否願意即時再開一局；不預先設定「好玩」結論。

如理解度未達 Gate，先修正規則表達或 UI，唔增加更多武將與關卡。
